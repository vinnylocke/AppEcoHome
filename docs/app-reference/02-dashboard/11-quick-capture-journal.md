# Quick Capture Journal

> Phone-first capture-first journal mode. Snap a photo + jot a note → Save. The entry lands in `plant_journals` without an `inventory_item_id`, then you assign it to a specific plant later from either device. Backed by the same `plant_journals` table the [Global Journal](../03-garden-hub/11-global-journal.md) and the per-instance [Plant Journal Tab](../08-modals-and-overlays/10-plant-journal-tab.md) read — no migration, no parallel schema. Unassigned entries surface on the Global Journal under the "Unassigned" filter; once assigned, they appear on the relevant plant's journal tab too.

**Route:** `/quick/journal`
**Source files (entry points):**
- `src/components/quick/QuickCapture.tsx` — the screen
- `src/components/quick/AssignToPlantSheet.tsx` — bottom-sheet plant picker
- `src/hooks/useUnassignedJournals.ts` — data hook (entries / refresh / assign / remove)

## Quick Summary

**Wave 6 focus-mode chrome**: on mobile, the persistent top bar + side nav are hidden by App.tsx (`isFocusMode`); the `QuickAccessMenuButton` floats top-right and opens the `MobileNavDrawer` on tap. The back chrome ("← Quick" + "Quick Capture" label) sits top-left and never collides with the menu button.

A focused mobile screen with two halves stacked vertically:

1. **Composer (top)** — a `PhotoUploader` + a textarea + a Save button. Save is enabled when *either* a photo or a description is present. On Save, inserts a `plant_journals` row with `inventory_item_id = NULL` and an auto-generated subject like `"Capture · 20 May, 14:32"` (editable inline later via the Plant Journal Tab once assigned).
2. **Recent Captures (bottom)** — unassigned `plant_journals` rows for this home, newest first. Each row shows the thumbnail, subject, snippet, and timestamp; the Assign affordance opens `AssignToPlantSheet` to pick a plant from the user's Shed.

Desktop visitors see the same screen with a small "mobile shortcut" banner — captures + assignments work identically there, which is how users can capture on a phone and file on a desktop later.

## Role 1 — Technical Reference

### Component graph

```
QuickCapture (mounted at /quick/journal)
├── Back chrome (chevron-left "Quick" + "Quick Capture" label)
├── Desktop banner (when useIsMobile() === false)
├── Composer
│   ├── PhotoUploader (bucket "plant-images", prefix "plant-photos") — same uploader the Plant Journal Tab uses
│   ├── Description textarea (optional)
│   └── Save button (enabled when photo OR description present)
├── Recent Captures list (data from useUnassignedJournals)
│   └── Per row
│       ├── Thumbnail or fallback sprout icon
│       ├── Subject + description snippet + relative timestamp
│       ├── "Assign →" → AssignToPlantSheet
│       └── Delete (trash) → confirm modal
├── AssignToPlantSheet (portal, when assigningEntry !== null)
└── Delete-confirm modal (portal)
```

### Props received

| Prop | Type | Source | Purpose |
|------|------|--------|---------|
| `homeId` | `string` | App.tsx | Scope; threaded through to `useUnassignedJournals`, every insert/update/delete, and the `AssignToPlantSheet`'s inventory query. |

### State (local)

| State | Purpose |
|-------|---------|
| `description` | Composer textarea content. |
| `imageUrl` | URL returned by `PhotoUploader` after a successful upload. |
| `uploading` / `saving` / `deleting` | In-flight flags for disabling buttons. |
| `assigningEntry` | The entry currently open in the Assign sheet, or `null`. |
| `pendingDeleteId` | The entry queued for the delete-confirm modal, or `null`. |
| `canSave` | Derived: `!!imageUrl || description.trim().length > 0`. |

### Data flow — read paths

```ts
// useUnassignedJournals(homeId)
supabase.from("plant_journals")
  .select("id, subject, description, image_url, created_at")
  .eq("home_id", homeId)
  .is("inventory_item_id", null)
  .order("created_at", { ascending: false })
  .limit(100);
```

`AssignToPlantSheet` makes its own one-time read of the home's inventory:

```ts
supabase.from("inventory_items")
  .select("id, plant_name, status, plants(common_name), areas(name, locations(name))")
  .eq("home_id", homeId)
  .neq("status", "Archived")
  .order("plant_name", { ascending: true })
  .limit(500);
```

### Data flow — write paths

| Action | DB call | Notes |
|--------|---------|-------|
| Save capture | `supabase.from("plant_journals").insert({ home_id, inventory_item_id: null, subject: autoGenerated, description, image_url })` | Idempotent at the row level — re-clicking Save creates another row, which is fine for "snap several plants in a session". |
| Assign capture | `supabase.from("plant_journals").update({ inventory_item_id }).eq("id", entryId)` (via the hook). Optimistic local removal from the unassigned list. | RLS `"Users can manage journals for their home"` permits this; the existing Plant Journal Tab will pick the row up when the user opens that plant. |
| Delete capture | `supabase.from("plant_journals").delete().eq("id", entryId)` | Confirmation modal first. |

### Edge functions invoked

None. The screen reads + writes the existing `plant_journals` table directly through Supabase's client.

### Cron / scheduled jobs

None.

### Realtime channels

None. The screen refreshes locally after every write; cross-device sync happens on the next page-load.

### Tier gating

None. Available on every tier — the journal is core to the app, not an AI feature.

### Beta gating

None.

### Permissions

Standard `plant_journals` RLS: home members can SELECT / INSERT / UPDATE / DELETE rows for homes they belong to. No additional permission keys.

### Error states

| State | Result |
|-------|--------|
| Insert fails (RLS, network) | Toast with the message, composer stays populated |
| Assign update fails | Toast inside the sheet; sheet stays open for retry |
| Delete fails | Toast; confirm modal closes |
| `useUnassignedJournals` query fails | Inline error banner in the Recent Captures section |

### Performance

- `useUnassignedJournals` caps the list at 100 entries with a single indexed query.
- `PhotoUploader` client-compresses to 1600px JPEG @ 85% before upload (existing helper).
- Lazy-loaded via `lazy(() => import(...))` in `App.tsx`.

### Linked storage buckets

- `plant-images` (path prefix `plant-photos`) — shared with the desktop Plant Journal Tab so users can re-use the existing photo timeline view without re-uploads.

## Role 2 — Expert Gardener's Guide

### Why open this screen

You've spotted something you want to remember — yellow spots, a flower opening, a label that fell off, a pest. Right now, in the garden, you don't have time to find the exact plant in your Shed, scroll to its journal tab, and pick the right one. Quick Capture removes that friction: you snap, you scribble a note, you save. Later, sitting on the sofa or at your desk, you assign each capture to the plant it was about.

The capture is timestamped automatically, so even if you forget which plant a note was for, the photo + the time + your description usually triangulate it.

### Every flow on this screen

#### 1. Snap or upload a photo

- **What you see**: a photo dropzone with a camera icon.
- **What you do**: tap → choose Take Photo (Capacitor native) or Choose from Library.
- **What happens next**: photo uploads to the `plant-images` bucket. The dropzone shows a thumbnail.

#### 2. Write a note

- **What you see**: a textarea labeled "Jot a note — e.g. 'yellow spots on the lower leaves'."
- **What you do**: type whatever you'd say to a fellow gardener. No formatting, no titles.
- **What happens next**: nothing yet — the note is in your composer.

#### 3. Save

- **What you see**: a Save button at the bottom. Disabled until you have *either* a photo or a note.
- **What you do**: tap Save.
- **What happens next**: the capture lands in `plant_journals` with `inventory_item_id = NULL`. The composer clears and your new entry appears in the Recent Captures list below.

#### 4. Assign to a plant (later, from any device)

- **What you see**: each Recent Captures row has an "Assign →" affordance.
- **What you do**: tap → bottom sheet opens with a search box + the list of your home's planted inventory.
- **What happens next**: pick a plant → sheet closes → entry leaves the Recent Captures list (it's now visible inside that plant's Journal tab in the Instance Edit Modal).

#### 5. Delete a capture

- **What you see**: a trash icon on each unassigned entry.
- **What you do**: tap → confirmation modal → Delete.
- **What happens next**: the row is permanently removed. Photo storage isn't garbage-collected automatically (a sweep job could be added later if it becomes an issue).

### Information on display — what every field means

| Element | Meaning |
|---------|---------|
| Photo dropzone | Tap to take a photo or upload one. Max 5MB. Compressed to 1600px before upload. |
| Description textarea | Free text. Newlines preserved. Save without one is fine if you have a photo. |
| Save button | Enabled when at least one of: photo, description (after trimming whitespace). |
| Recent Captures count | Number of unassigned captures for this home. |
| Capture subject | Auto-generated `"Capture · {day} {month}, {hour}:{minute}"`. |
| Timestamp | When you saved the capture, in your local time. |

### Tier-by-tier experience

No differences. Available on every tier.

### New user vs returning user vs power user

- **Brand new user**: empty state in the Recent Captures section nudges them to take their first capture.
- **Returning user with a few unassigned**: list is the to-do list. Tap Assign on each one as you have time.
- **Power user**: rapid-fire snap-snap-snap in the garden, then bulk-assign on the desktop later. The list is unbounded (cap is 100; we'll add Load More if anyone hits it).

### Beta user experience

No difference.

### Common mistakes / pitfalls

- **Saving with no photo or no note**: the Save button is disabled until you have one. Nothing to fix here, but worth noting.
- **Letting captures pile up unassigned**: each one is fine on its own, but if you wait six months you'll have a hard time remembering which plant each one was about. Aim to clear the list weekly.
- **Expecting the AI to identify the plant for you here**: Quick Capture is a notes surface, not an identification surface — that's what Visual Lens (`/quick/lens`) is for. Use both.

### Recommended workflows

- **In-the-garden inspection**: walk the beds, snap anything interesting + 5-second voice-to-text note, Save. File later.
- **Weekly review**: open `/quick/journal` on your desktop, sit down with coffee, work through the list one row at a time — Assign each to the right plant.
- **Bug detected**: snap → "thrips on the broad bean leaves" → Save. Assign to the broad bean later, then run Visual Lens on the same plant for AI-confirmation + treatment tasks.

### What to do if something looks wrong

- **Capture missing from the list**: it may have been assigned already (check the plant's Journal tab) or deleted from another device. Refresh the page.
- **Photo uploaded but capture didn't save**: rare, but the photo lives in storage until next sweep. Re-write your note and Save — pick the new photo from the library if needed.
- **Wrong plant assigned**: open the plant's Journal tab and delete the entry, then re-capture and re-assign. (Edit-in-place from `/quick/journal` after assignment isn't supported — that's the Plant Journal Tab's job.)
