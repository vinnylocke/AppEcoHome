# Sowing Calendar ↔ Nursery Bridge

## Goal

Two directions of integration between the existing **task system** and the existing **Nursery** (seed packets + sowings):

### Direction A — Task completion → Nursery sowing

When a user creates or completes a "Planting" task that is **explicitly linked to a seed packet**, automatically create a matching `seed_sowings` row against that packet on completion. No more "I sowed those seeds — let me also remember to log it in the Nursery."

### Direction B — Packet → Sowing calendar

When a packet exists in the Nursery, surface a **sowing calendar tab** on the packet detail showing:
- The recommended sowing windows (sow indoors / sow direct / transplant out) — pulled from the existing `grow_guides` propagation + germination sections
- Calendar-style timing visualisation respecting the user's hemisphere
- A "Set up sowing schedule" CTA that adds the relevant tasks to the calendar, **pre-linked to this packet** so Direction A then kicks in automatically when the user completes them

## Investigation summary

What's already in place:

- **`grow_guides` table** has `propagation` + `germination` sections per plant, each with a `schedulable_tasks: SchedulableTask[]` field — the AI already generates these on grow-guide creation. Cached per `plant_id`, generated by `generate-grow-guide` (or via `plant-doctor` action).
- **`AddToCalendarSheet`** ([src/components/growGuide/AddToCalendarSheet.tsx](src/components/growGuide/AddToCalendarSheet.tsx)) already converts schedulable_tasks → blueprint rows + per-instance picker + duplicate detection. We reuse it.
- **`NurseryPacketPicker`** ([src/components/nursery/NurseryPacketPicker.tsx](src/components/nursery/NurseryPacketPicker.tsx)) — reusable Shed-aware packet picker, already used by AddTaskModal in some flows.
- **`nurseryService.logSowing`** writes a `seed_sowings` row given `{ packet_id, sown_on, sown_count, notes }`.
- **Hemisphere-aware seasonal logic** in [src/lib/seasonal.ts](src/lib/seasonal.ts).

What's missing:

- `tasks` and `task_blueprints` don't carry `seed_packet_id` — no way to associate a Planting task with the packet it's sowing from.
- No "Sowing calendar" surface on the packet detail modal.
- No bridge between "Planting task completed" and "Sowing logged in the Nursery".
- Grow guide schedulable tasks have a `task_type` but don't carry a window date — they describe activities ("Sow seeds indoors") without committing to a calendar month. We need to render those windows in a hemisphere-aware way for visual presentation.

## Sensible-default decisions

| Decision | Choice |
|---|---|
| Where to put the packet link on tasks | **`tasks.seed_packet_id` + `task_blueprints.seed_packet_id`** as nullable FKs (ON DELETE SET NULL). Cleanest: the task already has `inventory_item_ids` for plant linking; this is the analogous link for the seed-side. |
| How to surface the link in the UI | **Show a "Link to a seed packet" picker inside AddTaskModal ONLY when `type === "Planting"`.** Hidden for other categories — most Planting tasks aren't sowings, so the picker is optional. |
| When task completion creates a sowing | **Only when `tasks.seed_packet_id IS NOT NULL`.** Avoids creating ghost sowings from "plant out the seedlings" tasks. Idempotent: a unique partial index on `seed_sowings(task_id) WHERE task_id IS NOT NULL` prevents duplicates from uncomplete-then-recomplete cycles. (Yes — we'll also add `seed_sowings.task_id` as a nullable FK.) |
| What `sown_count` defaults to on auto-create | **`1` (best-effort starter).** The user can edit any time via the existing Observe / Edit Sowing flow. We prompt for the exact count in a small toast follow-up ("Sowing logged — set the count?") with a deep link. |
| Sowing calendar data source | **The existing `grow_guides` row for the packet's `plant_id`** — propagation + germination sections only. If no guide exists yet, the calendar UI surfaces a "Generate sowing calendar" CTA that fires the existing `generate-grow-guide` (Sage+ only) or shows a tier-gated nudge. Avoids per-packet Gemini calls; same plant → same calendar. |
| What if the packet has no linked plant_id | **Show an empty-state**: "Link a plant to this packet to see its sowing calendar." Already a common state for bulk-paste packets. Direct link to the existing EditSeedPacketModal "link plant" flow. |
| Hemisphere awareness of calendar | **Always** — the calendar header reads "Northern hemisphere · Spring sowings" / "Southern hemisphere · Autumn sowings" and reorders months accordingly using existing `seasonal.ts` helpers. |
| Tier gating | **Free for everyone** — calendar reading + linking tasks to packets + auto-creating sowings. The ONLY tier-gated piece is the initial `generate-grow-guide` call if the user wants AI-generated content (already gated today; no new gate). |
| Auto-update journal compatibility | **The auto-created sowing carries `task_id` AND the user's existing `auto_update_journal_categories` preference still fires.** End result for an opt-in user: completing a sow task → (1) Nursery sowing row created, (2) journal entry created tagged to the plant instance (when known) or unassigned. Both already idempotent on `task_id`. |
| Auto-update visibility | **Soft-confirm toast** on creation: "Sowing logged in the Nursery · Open" — gives the user the option to set the count or observe germination immediately. No popup interrupt. |
| Where the "Sowing calendar" tab lives | **Inside SeedPacketDetailModal as a new tab strip.** The modal currently shows Sowings + Edit + Archive — adding a "Calendar" tab fits the same modal without a new surface. |
| Visual rendering | **A horizontal 12-month strip** (responsive to year), hemisphere-aware month order, with coloured bands per activity: Sow indoors (green), Sow direct (amber), Transplant out (blue). Each band is clickable → opens AddToCalendarSheet pre-filled with that task. |
| Do we add a "Sowing" task category | **No.** Keep TASK_CATEGORIES as-is. The differentiator is `tasks.seed_packet_id` being set — semantically richer than a category split, and avoids touching every place that filters by category. |

## Schema changes

```sql
-- Migration 1: link tasks (and their blueprints) to seed packets.
alter table public.tasks
  add column if not exists seed_packet_id uuid
    references public.seed_packets(id) on delete set null;

alter table public.task_blueprints
  add column if not exists seed_packet_id uuid
    references public.seed_packets(id) on delete set null;

create index if not exists tasks_seed_packet_idx
  on public.tasks(seed_packet_id) where seed_packet_id is not null;
create index if not exists task_blueprints_seed_packet_idx
  on public.task_blueprints(seed_packet_id) where seed_packet_id is not null;

-- Migration 2: link sowings back to the originating task so we can
-- enforce idempotency and surface "this sowing came from task X" later.
alter table public.seed_sowings
  add column if not exists task_id uuid
    references public.tasks(id) on delete set null;

create unique index if not exists seed_sowings_task_id_unique
  on public.seed_sowings(task_id) where task_id is not null;
```

No data migration needed — all new columns are nullable defaults.

## Decisions to confirm before implementation

1. **Add `seed_packet_id` to BOTH `tasks` and `task_blueprints`** (so recurring "weekly direct sowings" blueprints also link)? Or just `tasks` for v1?
2. **Auto-create `sown_count = 1`** as a placeholder with a "set count" toast nudge, or **prompt the user** with a small inline modal at completion time?
3. **Calendar bands**: 3 colour-coded activities (sow indoors / sow direct / transplant out) — happy with that taxonomy, or want a different split (e.g. add "Pot on" / "Harden off")?
4. **Sowing calendar empty state for packets without a linked plant**: link straight to the EditSeedPacketModal's "link plant" section (current plan), or offer an inline "Generate from variety name" Gemini path (more friction, more flexibility)?
5. **The new tab strip in SeedPacketDetailModal** — happy with "Sowings | Calendar" as the two tabs (Calendar selected by default when packet has both a plant_id AND no active sowings)? Or always default to Sowings?

Once those are confirmed, implementation begins in the order above.
