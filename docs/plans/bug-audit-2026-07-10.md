## CRITICAL

### 1. `[x]` custom_http water valve can be left stuck open — **FIXED (local, pending deploy) 2026-07-10**
- **Fix:** new shared dispatcher `_shared/integrations/valveControl.ts` `controlValve(...)` routes adapter providers (custom_http) through `adapter.control()` and non-adapter providers (eWeLink) through the caller's fallback. Applied to all **three** eWeLink-hardcoded paths: `valveQueue.ts` (queue drain), `run-automations` `fireValves` (direct/manual fire — a third copy found during the fix), and `integrations-dead-mans-switch` (backstop). eWeLink behaviour unchanged (null adapter → fallback). Deno tests `valveControl.test.ts` VC-001..006 (incl. real-registry invariants); full suite 929 green; both changed edge fns have zero new type errors vs HEAD. App-reference `09-data-model-integrations.md` updated.
- **Where:** `supabase/functions/integrations-dead-mans-switch/index.ts:22,83-99` (backstop) + `supabase/functions/_shared/valveQueue.ts:185-232` (primary off-path) + `run-automations` `fireValves`.
- **What:** The 60-second dead-man's-switch — the safety net that force-closes any valve whose `auto_off_at` has passed — hardcodes `EWELINK_BASE` and always builds an eWeLink `switch:off` payload (`/v2/device/thing/status`). There is **no provider branch**. For a `custom_http` (DIY) valve: `decryptCredentials` yields no `accessToken`, `meta.direct_device_id` is undefined, the POST to `eu-apia.coolkit.cc` fails every tick → `success=false` → it re-arms `auto_off_at +5min` and retries the same doomed call forever. The valve is never closed by software.
- **Why CRITICAL:** this is real irrigation. eWeLink valves have layered auto-off (device countdown + queued turn_off + this backstop), so they're safe. A `custom_http` valve relies solely on its own DIY firmware to self-close; if it doesn't, the app's two software backstops both silently no-op → water runs indefinitely (flooding / waste / drowned plants).
- **Blast radius:** users with a `custom_http` water valve. Manual control (Garden Walk / `integrations-adapter-control`) uses the correct adapter and works — it's the **automation** and **backstop** paths that are eWeLink-only.
- **Fix direction:** both paths must branch on `device.provider` / `integration` type and dispatch through the same adapter registry the manual path uses (`_shared/integrations/adapters/*`), not a hardcoded eWeLink POST.

### 3. `[x]` `generate-daily-brief` unauthenticated regeneration — **FIXED (local, pending deploy) 2026-07-10**
- **Fix:** any targeted `{ homeId }` call (regenerate or not) now requires `requireAuth` + `requireHomeMembership` before generation; regenerate keeps its extra Sage+ tier + rate-limit. The no-body `{}` cron sweep stays open (standard `verify_jwt=false` cron path). Verified live: local `{homeId}` no-auth now returns 401 (was 200). Tests EF-008/009/014.
- **Where:** `supabase/functions/generate-daily-brief/index.ts:226-243`; `config.toml` `verify_jwt=false`.
- **What:** requireAuth + membership + tier + rate-limit all live inside `if (regenerate)`. The `{ homeId }` path (no `regenerate` flag) skips the whole block and runs full generation + Gemini voice + upsert of the victim's `daily_briefs` row. `verify_jwt=false` means no JWT is needed either.
- **Verified live:** `POST /generate-daily-brief {}` with only the public publishable key → **200** `{"success":true,"homes":4,"generated":4}`. The anon key ships in the browser bundle, so this is internet-reachable.
- **Impact:** unauthenticated Gemini cost burn (loopable) + arbitrary overwrite of any Sage/Evergreen home's brief.
- **Fix direction:** move `requireAuth` + membership above the branch; require it for the on-demand `{homeId}` path too (cron uses the no-body path which is fine behind the key).

### 4. `[x]` `generate-grow-suggestions` on-demand path unauthenticated — **FIXED (local, pending deploy) 2026-07-10**
- **Fix:** the `{ homeId }` branch now requires `requireAuth` + `requireHomeMembership`; the `{}` cron sweep stays open. Test EF-010.
- **Where:** `supabase/functions/generate-grow-suggestions/index.ts:58-65`.
- **What:** `{homeId}` → `homeIds=[homeId]` with no `getUser()`/membership check; `{}` → every home. Identical to the generate-pest-risk hole that was already fixed on 2026-07-02 (its fix at `generate-pest-risk:55-72` is the template). Evergreen-gated AI, so this is paid-feature cost burn on any home.

### 5. `[ ]` Personal recurring tasks leak home-wide (scope drop)
- **Where:** `src/lib/taskMutations.ts:8-29` (`buildGhostPayload`), `supabase/functions/generate-tasks/index.ts:118-134`, `src/components/TaskList.tsx` ghost inserts.
- **What:** Both the client materialisation payload and the nightly cron omit `scope`, `created_by`, `assigned_to` (and the cron also omits `plan_id`). A blueprint with `scope='personal'` carries these on its ghost, but the insert drops them → the row takes the DB default `scope='home'`, `created_by=NULL`. The scope-aware RLS SELECT policy then shows one member's personal task to the whole home, and it drops out of that member's "Mine" filter. The cron leaks the next 7 days of every personal routine nightly; plan-linked routines also vanish from plan views (`plan_id` lost).
- **Fix direction:** carry `scope`, `created_by`, `assigned_to`, `plan_id` through `buildGhostPayload` and the cron insert.

### 6. `[ ]` Weather rules evaluate the wrong "today" in non-UTC homes
- **Where:** `supabase/functions/analyse-weather/index.ts:116,139` (+ consumers at 223/278), `weatherTaskCreation.ts`, `_shared/weatherForecast.ts:60,71`, `_shared/weatherRules/frostRisk.ts:23`.
- **What:** `sync-weather` fetches Open-Meteo with `timezone:"auto"`, so `snapshot.daily.time[]` / `hourly.time[]` are **home-local** dates. `analyse-weather` computes `today = new Date().toISOString().split("T")[0]` — **UTC**. In non-UTC homes the two disagree for part or all of the day (worst when the `0 1 * * *` UTC cron fires in the previous local day for the Americas): `rainAutoComplete` reads *tomorrow's* forecast as today's rain and can auto-complete watering the evening before; heat `taskCreates` stamp the wrong `due_date`/claim key; the hourly frost filter drops tonight's remaining pre-midnight hours; frost banners display the hour offset by the UTC delta.
- **Note:** mostly benign for UK/EU (small offset); real day-shift for the Americas / APAC. This is a *new* cluster distinct from the tz bugs fixed on 2026-07-02.
- **Fix direction:** derive `today` from the home's timezone (snapshot carries it, or derive from lat/lng) rather than UTC; compare local-naive strings consistently.

### 7. `[ ]` Care-adjustment Apply not idempotent → double-apply
- **Where:** `src/lib/careAdjustments.ts:40-93`; brief card `GardenBrainBriefCard.tsx`, `AdaptiveCareCard.tsx:97-108`.
- **What:** `applyCareAdjustment` performs the mutation (blueprint update or create-routine) then sets `status='applied'` with `.eq("id", …)` only — **not** `.eq("status","proposed")`. The two surfaces render the same proposal on the same screen; the brief's inline Apply doesn't refresh the card's list. Apply on the brief, then Apply on the still-visible card → the create path runs twice → two watering blueprints + two first tasks + two `generateBlueprintTasks` runs. (Card→brief is guarded by `fetchCareAdjustment`; brief→card and simultaneous taps are not.)
- **Fix direction:** make the status flip a conditional CAS (`.eq("status","proposed")`) and gate the mutation on it having claimed the row; refetch/rerender the sibling surface on apply.

### 8. `[ ]` Ecowitt webhook cross-writes readings between homes
- **Where:** `supabase/functions/integrations-ecowitt-webhook/index.ts:96-104`.
- **What:** With a single global `ECOWITT_WEBHOOK_SECRET`, the device lookup is `.eq("device_type","soil_sensor").eq("provider","ecowitt")` then `.find(metadata.channel === ch)` — **no home/integration/MAC scope**. Two homes each with a channel-1 soil sensor collide: whichever device the unordered query returns first receives both homes' readings, driving the wrong home's sensor automations.

### 9. `[ ]` Custom-HTTP valve control follows redirects → SSRF
- **Where:** `supabase/functions/_shared/integrations/adapters/customHttp.ts:481`; validator `urlSafety.ts:52`.
- **What:** `checkControlUrl` validates only the initial hostname; `fetch(controlUrl, …)` uses the default `redirect:"follow"`. A user-supplied `control_url` that 302s to `http://169.254.169.254/…` reaches cloud metadata — the per-hop host is never re-checked.
- **Fix direction:** `redirect:"manual"` and re-run the host allow-list on every hop (or block redirects entirely for control URLs).

### 10. `[x]` `predict-yield` IDOR / cross-home disclosure — **FIXED (local, pending deploy) 2026-07-10**
- **Fix:** `requireAuth` + `requireHomeMembership(home_id, caller)` before any read, plus a check that the instance's `home_id` equals the supplied `home_id` (blocks a member of home A pairing A's id with B's instance). Test EF-011.
- **Where:** `supabase/functions/predict-yield/index.ts:24-72`.
- **What:** Service-role client reads `inventory_items` by `instance_id`, `yield_records` by `instance_id`, weather by `home_id`. `guardAiByHome(home_id)` only checks the *owner's* tier; `enforceRateLimit` limits the *owner*. There is **no** check that the caller is a member of `home_id`, nor that `instance_id` belongs to `home_id`. A caller passing another home's `instance_id` + `home_id` gets that plant's nickname, planting dates and yield history back in the prediction.

### 11. `[ ]` Completed overdue task vanishes instead of showing "completed today"
- **Where:** `src/lib/taskEngine.ts:265`.
- **What:** `const timestamp = task.updated_at || task.created_at || task.due_date` — but `tasks` has **no `updated_at` column** (select is `*`, so it's simply `undefined`). A task due before the visible window, completed today, resolves its completed-date from `created_at` → `isCompletedInWindow=false` → filtered out of Today the instant it's ticked; the RHO-19 "completed late" chip only shows on the old due-day agenda. `completed_at` exists and is the correct field. This is the same *class* the user has repeatedly hit ("it disappeared" / wrong counts).

### 21. `[ ]` "Neglected plant" false positive (user-reported)
- **Where:** `supabase/functions/_shared/patterns/neglectedPlant.ts:16-27`.
- **What:** The detector's ONLY definition of "touched" is a `user_events` row with `event_type='task_completed'` whose `meta.inventory_item_ids` contains the plant. That misses every other form of care:
  - **Automation watering** — an auto-water writes `device_commands`(turn_on)/`valve_events` against the *device's area*, and `device_readings`; it never emits a `task_completed` user event. A plant watered daily by an automation registers zero activity.
  - **Auto-completed tasks** — the rain/weather auto-complete (`analyse-weather`/`rainAutoComplete`/`fanoutActions`) and harvest/window auto-complete set `tasks.status='Completed'` server-side (service role) with no user event.
  - Any completion that didn't route through `taskActions.completeTask`/`TaskList` (the only two `task_completed` emitters).
- **Reproduces the report exactly:** strawberries on a daily watering automation + a completed pruning task, still flagged "untouched 14 days" — the automation is invisible and the completion likely came via an auto/bulk path.
- **Fix direction:** redefine "touched" from authoritative state within the 14-day window, home-scoped, ANY of:
  1. `tasks` where `status='Completed'` AND `completed_at >= cutoff` AND `inventory_item_ids` overlaps the plant (catches manual + auto + bulk + window completions).
  2. Watering activity on the plant's `area_id` — `device_commands`(turn_on) / `valve_events` joined via `devices.area_id = inventory_items.area_id` within the window.
  3. Recent `plant_journals` entries for the item (a photo/note is engagement).
  A plant is neglected only if planted ≥14 days ago AND none of the above. Add a Deno test (automation-watered → not flagged; auto-completed task → not flagged; genuinely idle → flagged) and update `26-pattern-engine.md`.

## Remediation sequencing

The findings fall into coherent PR-sized batches. Each batch is one plan doc + tests + app-reference updates + its own deploy, per the repo workflow. Proposed order (safety → security → correctness → reliability):

- **Batch 1 — Valve safety (CRITICAL #1, HIGH #2).** Provider-dispatch the automation off-path (`valveQueue.ts`) and the dead-man's switch through the adapter registry the manual path already uses, so a `custom_http` valve actually closes. Deno test for both providers. Ship first, alone.
- **Batch 2 — Edge-auth family (HIGH #3, #4; MED #10, #12, #13, #14).** Hoist `requireAuth` + membership above the on-demand branches; make `guardAiByHome` fail closed and check the *caller's* membership + tier; add rate limits to `add-plant-to-library`. One sweep across the named functions; mirrors the 2026-07-02 pest-risk fix. Deno auth tests.
- **Batch 3 — Scope/privacy (HIGH #5).** Carry `scope`/`created_by`/`assigned_to`/`plan_id` through `buildGhostPayload` + `generate-tasks`. Unit + Deno regression tests.
- **Batch 4 — Weather timezone (HIGH #6).** Derive `today` from the home timezone in `analyse-weather` + forecast/frost helpers. Deno tests for a non-UTC home.
- **Batch 5 — Garden Brain + pattern engine (HIGH #7; MED #16, #19, #21).** CAS the care-adjustment apply on `status='proposed'`; guard the `photo_observations.actions` jsonb write; key the cooldown off dismissal time; broaden the neglected-plant signal. This release's code — cheap to harden now.
- **Batch 6 — Devices/integrations (HIGH #8, #9; MED #18).** Scope the Ecowitt webhook device lookup by integration/MAC; block redirects in custom-HTTP control; bound `recorded_at` to now.
- **Batch 7 — Reliability `.error` workstream (MED #15, #17, #20 + the systemic theme).** A shared checked-mutation helper applied across the ~20 unchecked sites, with rollback on the optimistic paths.

Batches are independent and can ship in any order; the sequence is by risk. #11 (taskEngine `updated_at`) is a one-line fix that can ride with Batch 3 or 5.

### Follow-up (tracked separately) — cron fleet-sweep authentication
Every `verify_jwt=false` cron function (~15 of them: `pattern-scan`, `pattern-evaluate`, `generate-*` fleet paths, `garden-manager-report {cron:true}`, `daily-batch-notifications`, the `refresh-*` family, etc.) is reachable by anyone holding the public bundle key. For most this is harmless (idempotent, cheap). For the **expensive AI fleet paths** — `garden-manager-report {cron:true}` (#12), `generate-grow-suggestions {}`, `generate-daily-brief {}` — it's a cost-amplification vector. Closing it properly needs a **shared cron secret** applied uniformly (a `CRON_SECRET` env via `supabase secrets set`, passed by the cron `net.http_post` and checked by each fleet function) — a cross-cutting change to the cron migrations + fleet functions, done in one careful pass rather than piecemeal. Batch 2 closed every **user-reachable targeted-home** vector; this follow-up closes the **fleet** vector.

## Systemic theme — unchecked `supabase-js` `.error`

supabase-js **never throws** on a query error; it returns `{ data:null, error }`. Across the app, ~20 mutation/query sites `await` the builder and ignore the result, then either toast success or optimistically update state. Confirmed instances (beyond those above):

- `TaskList.tsx` bulk complete/postpone/delete (`Promise.all` of unchecked updates; the postpone path can **tombstone a task with no replacement** on a failed insert) — HIGH within this class.
- `PlannerDashboard.tsx:225-246` plan delete: unchecked tasks/blueprints cleanup then "Plan deleted" — orphaned blueprints keep firing.
- `AddTaskModal.tsx` dependency insert + online ghost-target insert (no 23505 fallback → duplicate task on retry).
- `usePushNotifications.ts` token upsert (silent → no notifications, nothing in Sentry).
- `useNotes.ts` update/delete, `saveToShed.ts` schedule inserts, `TaskList` planting-status update, `GardenerProfile` weather-toggle (RLS-filtered 0-row UPDATE toasts success for non-admins), `AutomationSuggestions`/`AiInsightsPage`/`ManagerLog` status flips.

**Recommendation:** treat this as one workstream — a small `mutate()` helper that checks `.error`, surfaces a toast, and (for optimistic paths) rolls back — applied across these sites. It's the highest reliability-per-effort fix in the audit.
